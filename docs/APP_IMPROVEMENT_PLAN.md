# App Improvement Plan — page-by-page & component-by-component

_Generated 2026-07-14 from a 15-area audit of the standardized Vela admin app (216 improvements). Build WITHIN the Vela design system: semantic tokens, `src/components/ui-app/*` primitives, `src/components/ui/spinner.tsx`, `src/lib/anim/*` GSAP hooks, lucide icons._

**Totals:** 65 high · 96 medium · 55 low across 15 areas.

---

## Dashboard page + widgets (src/pages/Index.tsx, src/components/dashboard/*)

_A functionally rich dashboard that is being dragged down by a bespoke StatCard duplicating the canonical token-based primitive, pervasive raw-palette colors that break in dark mode (QuickActions, ProfileStats, TopProducts ranks), a redundant auth/profile fetch waterfall, mixed animation runtimes (framer-motion vs the standard GSAP), and no count-up/entrance motion despite the anim system being available._

- **[HIGH · modern]** Index.tsx line 7 + StatCard.tsx (whole file); dashboard/StatCard.tsx
  - Issue: The dashboard imports the bespoke `@/components/dashboard/StatCard`, whose colorMap (lines 17-22) hardcodes raw palette + light-only borders: `bg-blue-500/10 text-blue-600 border-blue-200/50`, `emerald`, `violet`, `amber`. These are not tokens and the `border-*-200/50` borders are near-invisible/wrong in dark mode. The canonical `ui-app/StatCard` already exists with token-based tone tiles (bg-success/10 text-success, etc.), dark-safe, plus built-in count-up.
  - Fix: Delete dashboard/StatCard.tsx and import { StatCard } from '@/components/ui-app'. Swap the four call sites (Index.tsx 429-456) from color="emerald|blue|violet|amber" to tone="success|info|brand|warning" (legacy `color` also still maps). Removes a divergent duplicate and fixes dark-mode color.
- **[HIGH · modern]** QuickActions.tsx lines 129,137,145,153,161 (colorClass) + 122,132,140,148,156 (icon colors)
  - Issue: Every quick-action tile hardcodes raw light-only palette: `bg-blue-50 text-blue-600`, `bg-amber-50`, `bg-emerald-50`, `bg-pink-50`, `bg-violet-50`, and icons `text-blue-600/amber-600/emerald-600/pink-600/violet-600`. In dark mode `bg-*-50` renders as near-white blocks — visually broken. Bypasses the semantic token system entirely.
  - Fix: Replace with the same tone-tile recipe used in ui-app/StatCard's TILE map: `bg-info/10 text-info`, `bg-warning/10 text-warning`, `bg-success/10 text-success`, `bg-primary/10 text-primary` (pick a token for the pink 'Fix Images' — e.g. brand/primary). Drop the per-icon text color and let the tile's `text-*` cascade.
- **[HIGH · perf]** WelcomeHeader.tsx lines 16-28 (getUser + profiles fetch); ProfileStats.tsx lines 33-46
  - Issue: WelcomeHeader calls `supabase.auth.getUser()` (network round-trip) on mount, then a second `profiles` query for first_name, and gates the entire header behind `profileLoading`. AuthContext already exposes `user` instantly via `useAuth()` (resolved once, cached). ProfileStats ALSO fetches the same `profiles` row (first_name/last_name) separately — two redundant profile reads per dashboard load.
  - Fix: In WelcomeHeader use `const { user } = useAuth()` and read `user.user_metadata?.first_name` for an instant name (no getUser, no loading gate). Consolidate the profiles read into one place (or expose first/last name from ShopContext/AuthContext) so ProfileStats and WelcomeHeader don't both query it.
- **[HIGH · motion]** Index.tsx lines 429-456 (StatCard values passed as strings)
  - Issue: Stat values are passed pre-formatted as strings (`formatCurrency(...)`, `${data.salesCount}`, `.toString()`), so even after switching to ui-app/StatCard the numbers snap in with no animation. The card's useCountUp only runs when `value` is a number. Revenue/Sales/Products/Customers currently pop instantly on every load and date-range change.
  - Fix: Pass numeric values plus a formatter: `value={data.totalRevenue} formatValue={(n)=>formatCurrency(n, shopDetails?.currency)}`, and raw numbers for sales/products/customers. This activates the reduced-motion-aware count-up for a smoother, modern feel on load and when filters change.
- **[MEDIUM · motion]** Index.tsx render (401-507) — no entrance motion
  - Issue: After the skeleton, the whole dashboard (GetStarted card, stat grid, chart, right column) appears in one instant frame. The standardized GSAP `useReveal` (reduced-motion aware, staggered) is unused here even though sibling surfaces adopt the motion system.
  - Fix: Wrap the main content in a `useReveal()` ref and mark the stat cards + the two main-area columns with `data-reveal` for a subtle staggered rise-in. Keep it subtle (default DISTANCE/STAGGER) per the productivity-app guidance.
- **[MEDIUM · consistency]** ActivityFeed.tsx line 13 + 280-318; QuickActions.tsx line 18 + 38-48
  - Issue: ActivityFeed and QuickActions animate with framer-motion (`motion.button`, `whileHover`, `AnimatePresence`), but the app's standardized motion system is GSAP (src/lib/anim). This ships two animation runtimes and diverges from the design-system directive to use useGsap/useReveal.
  - Fix: Port the ActivityFeed row entrance to GSAP (useReveal on the list, keyed by activities.length) and drop framer-motion from these files. Standardizes on one motion runtime and trims bundle weight.
- **[MEDIUM · cleanliness]** Index.tsx lines 462-465, 482-484, 489-491, 498-500 vs OverviewChart.tsx 83 / TopProducts.tsx 87 / ActivityFeed.tsx 252
  - Issue: Each right/left panel gets a tiny muted section heading (icon + 'Business Overview' / 'Top Sellers' / 'Live Activity') AND the card inside renders its own `CardDescription` ('The performance of your business...', 'Best selling products', 'Click on an event...'). The result is a doubled, redundant header stack that eats vertical space in an already viewport-locked layout.
  - Fix: Pick one heading source. Move the icon+title into each card's CardHeader (replacing CardDescription) and drop the external `<h2>` rows, or keep the external headings and remove the CardDescription. Reclaims ~2 rows of height per panel.
- **[MEDIUM · usability]** Index.tsx line 402 `h-[calc(100vh-7rem)]` + right column 480-505 (three flex-1 basis-0 panels)
  - Issue: The whole dashboard is locked to viewport height, and the right column stacks Profile (fixed) + TopProducts (flex-1) + Activity (flex-1) inside it. On short laptop screens TopProducts and Activity get crushed to 1-2 visible rows each, and when GetStartedCard shows the stat grid + chart compress further. This also diverges from sibling pages that use PageContainer + natural scroll.
  - Fix: Drop the hard 100vh lock (or make it a min-height that releases below ~lg), give TopProducts/Activity sensible min-heights, and let the page scroll on smaller viewports. Consider adopting PageContainer for consistent outer padding/rhythm with other routes.
- **[MEDIUM · consistency]** OverviewChart.tsx lines 85-119 (bespoke segmented buttons) vs 123-131 (ToggleGroup)
  - Issue: The period selector (7D/30D/90D/6M/All) is a hand-rolled `<button>` segmented control with manual active-state styling, sitting directly above a shadcn ToggleGroup used for granularity and series. Two different segmented-control patterns side by side; the '6M' active check (`activePeriod === null && dateRange !== undefined`) is brittle.
  - Fix: Rebuild the period presets as a single-select ToggleGroup (variant="outline" size="sm") matching the granularity/series controls, driving one `activePeriod` value. Unifies the toolbar's look and removes the fragile active-state booleans.
- **[MEDIUM · consistency]** TopProducts.tsx lines 104-105 (empty) & ActivityFeed.tsx line 325 (empty); Index.tsx 381-399 (error / no-data)
  - Issue: Empty/zero states are bare centered `<p className='text-sm text-muted-foreground'>` strings (top sellers, activity feed, dashboard no-data). The error state (Index 384) uses raw `text-amber-500` instead of a token. The canonical EmptyState primitive (icon + title + copy + action) is unused, so these 'nothing here' moments feel unfinished and inconsistent.
  - Fix: Use `<EmptyState compact icon={...} title=... />` for TopProducts (e.g. Package icon, 'No sales yet') and ActivityFeed (Activity icon). For the dashboard error/no-data blocks use EmptyState (icon={AlertTriangle}) and swap `text-amber-500` → `text-warning`.
- **[MEDIUM · modern]** ProfileStats.tsx lines 110,116,122 (StatPill colorClass) + 74 (IG badge gradient)
  - Issue: The follower/posts/products pills hardcode raw palette tints `bg-blue-500/8`, `bg-violet-500/8`, `bg-emerald-500/8` instead of tokens. Not part of the semantic system and the /8 opacity is barely visible in dark mode.
  - Fix: Map the three pills to token tints (`bg-info/10`, `bg-primary/10`, `bg-success/10`) with matching `text-*` on the icon. The IG badge gradient can stay as an intentional brand mark.
- **[MEDIUM · modern]** TopProducts.tsx lines 23-27 (RANK_CONFIG)
  - Issue: Rank badges hardcode raw palette: #1 amber-400, #2 `bg-slate-400/10 text-slate-500`, #3 `bg-orange-400`. slate/orange are not in the semantic token set, and the whole-row `rank.bg` tinting on every item makes the list visually noisy.
  - Fix: Tokenize: keep a subtle gold-ish treatment for #1 via `warning`, use `neutral`/muted for #2 and #3, and consider tinting only the rank badge (not the entire row) so rows read cleaner.
- **[LOW · usability]** Index.tsx line 498-500 ('Live Activity' heading) + ActivityFeed realtime
  - Issue: ActivityFeed is fully realtime (orders/products/disputes channels) but nothing signals 'live' — the heading is a static Activity icon + text. Users don't know the feed auto-updates.
  - Fix: Add the design-system `<StatusDot tone="success" pulse />` next to the 'Live Activity' heading (or in the card header) to communicate the live/streaming state, matching the existing StatusDot pulse primitive.
- **[LOW · cleanliness]** QuickActions.tsx lines 30-62 (QuickActionCard) vs 178-193 (actual render)
  - Issue: `QuickActionCard` (a framer-motion lift/tap card) is defined but never rendered — the component maps `actions` to plain `<button>`s with only `transition-colors`. Dead code, and the rendered buttons have no hover lift/affordance.
  - Fix: Delete the unused QuickActionCard, or actually use it. If keeping the compact button row, add a subtle hover elevation (border/bg token shift) so the actions feel interactive and on-brand.
- **[LOW · cleanliness]** TopProducts.tsx lines 146-157 (img onError imperative DOM fallback)
  - Issue: The product thumbnail uses a raw `<img>` whose onError imperatively hides the element and injects a hand-written SVG via `innerHTML` into the parent — bypassing React, and duplicating the `ImageOff` icon that's already imported and used for the null-src case.
  - Fix: Replace with a small stateful `<ProductThumb>` that flips to the imported `ImageOff` (or a shared image component) on error, removing the innerHTML/createElement hack.
- **[LOW · usability]** Index.tsx lines 355-378 (loading skeleton) + QuickActions.tsx line 83 (window.location.reload)
  - Issue: Two smaller polish gaps: (1) the loading skeleton omits the GetStartedCard and the per-panel heading rows, so real content shifts layout when it loads; (2) 'Fix Images' calls `window.location.reload()` after refreshing media — a jarring full-page reload in an SPA.
  - Fix: Align the skeleton to the real structure (reserve heading rows), and replace the hard reload with a context refetch (e.g. re-run the products/media query or fetchShopDetails) so images update in place.

## Products page (src/pages/Products.tsx + ProductCard.tsx, ProductTableView.tsx, ProductTable.tsx, ProductStatusDropdown.tsx, BulkActionsToolbar.tsx, products/ProductFilterPanel.tsx, hooks/useProductData.ts, useProductFilters.ts)

_A dense, feature-rich page that mostly reads modern, but it leaks raw Tailwind palette colors (emerald/amber/red/gray/slate) in the card and status dropdown that break dark mode, has an N+1 variant-stock query and getUser waterfalls, and re-implements primitives (search, command bar, empty state, stat pills) instead of composing the ui-app/anim system, causing drift from sibling pages like Orders._

- **[HIGH · consistency]** ProductCard.tsx getStockDot + Set Price + star (lines 85-100, 108, 153, 211)
  - Issue: The stock dot reimplements stock coloring with RAW palette: text-emerald-600/bg-emerald-500, bg-gray-400, text-amber-600/bg-amber-500, text-red-500/bg-red-500 (lines 87-99). 'Set Price' uses text-amber-600 (211), the rating star fill-amber-400/text-amber-400 (153), and selection ring uses ring-gray-400 (107). These hardcoded colors do not adapt to dark mode and violate the token rule, so the card looks off-brand and washes out on dark. It also duplicates stock logic that already exists in lib/status.ts.
  - Fix: Replace with semantic tokens: text-success/bg-success, text-warning/bg-warning, text-destructive/bg-destructive, text-muted-foreground/bg-muted-foreground; ring-gray-400 -> ring-border. Better: derive the stock chip from stockTone()/stockLevel() in lib/status and render it via <StatusDot tone={stockTone(inventory)}/> from ui-app so it matches the table and filter panel. Star -> text-warning fill-warning.
- **[HIGH · consistency]** ProductStatusDropdown.tsx statusConfig (lines 18-22, 33-40)
  - Issue: Status colors are hardcoded raw palette (text-emerald-600 / text-amber-600 / text-slate-600) that break in dark mode, and the trigger is a bare ghost button with an icon+label but NO chevron/caret, so it reads as static text rather than an interactive status switcher. Users may not discover they can change status from the card.
  - Fix: Map status -> tone via productStatusTone() + toneText from lib/status (success/warning/neutral) instead of raw colors, and add a small ChevronDown (lucide) to the trigger plus a subtle hover bg so the affordance is obvious. Consider rendering the current state as a <StatusBadge> to match the table.
- **[HIGH · perf]** ProductTableView.tsx fetchVariantStockSummary (lines 60-122)
  - Issue: Variant stock is fetched with one Supabase query per product (products.forEach -> fetchVariantStockSummary -> select from product_variants where product_id = id). A 100-row table fires ~100 round-trips (classic N+1), spiking read load and causing staggered spinner flicker in the Variants column.
  - Fix: Batch a single query: select inventory, product_id from product_variants where product_id in (ids for the current page), then group client-side into per-product summaries and populate the cache in one pass. Keep the updated_at cache key for invalidation.
- **[HIGH · perf]** Products.tsx auth + realtime (lines 415, 444-448, 511; useProductData already uses useAuth)
  - Issue: The page calls supabase.auth.getUser() three separate times (fetchBusinessDataForSyncStatus, setupRealtimeListener, handleSync 'full') even though AuthContext exposes userId synchronously (useProductData already consumes it). This is the exact getUser waterfall the design guidance flags. It also opens a SECOND realtime channel on the products table (products:${user.id}) duplicating the subscription already in useProductData.
  - Fix: Pull userId from useAuth() and drop the getUser() awaits; gate the business-status effect on userId. Fold the debounced business-sync refresh into useProductData's existing products channel (or expose an onInsertBurst callback) so only one products subscription exists.
- **[HIGH · usability]** Products.tsx table branch (lines 979-990) / ProductTableView Table (line 172)
  - Issue: The table has 9 columns (checkbox, Image, Name, Status, Price, In stock, Variants, Total Earned, Actions) but is rendered raw inside Card>CardContent with no horizontal scroll container. On mid-width desktops the table overflows and scrolls the whole PAGE horizontally, which the layout should never do. The redundant 'In Stock/Low Stock' badge next to the product name (line 219) also duplicates the dedicated In-stock and Variants columns.
  - Fix: Wrap the table in a div with overflow-x-auto (or shadcn ScrollArea horizontal) so only the table scrolls. Drop the inline stock badge from the Name cell since In-stock + Variants columns already convey it, reducing column pressure.
- **[MEDIUM · motion]** Products.tsx grid stagger (lines 84-95, 960-974) + ProductCard motion.div layout (line 103)
  - Issue: Grid entrance uses framer-motion staggerChildren:0.05 with no cap, so a large catalog animates the last card ~ (N*0.05)s in (100 products = ~5s of cascading fade-in). Every card is also a motion.div layout, so any filter/resize triggers full layout animations across all cards = jank. This also diverges from the standardized GSAP anim system (useReveal) used elsewhere.
  - Fix: Cap the visible stagger (e.g. animate only the first ~12-16, or switch to useReveal from src/lib/anim with data-reveal on cards for a subtle, reduced-motion-aware entrance). Remove per-card layout animation (or scope it) so filtering doesn't reflow-animate the whole grid.
- **[MEDIUM · consistency]** Cross-file Draft tone: Products.tsx StatusPill (line 801), ProductCard/StatusDropdown (amber), ProductTableView (line 228-230 'warning') vs lib/status productStatusTone (Draft -> neutral) used by ProductFilterPanel
  - Issue: 'Draft' renders as amber/warning in the top status pills, the card status dropdown, and the table StatusBadge, but the canonical productStatusTone() maps Draft -> neutral, which the left ProductFilterPanel uses. So the same status shows as amber in three places and gray in the filter rail — a visible inconsistency within one page.
  - Fix: Pick one source of truth: either update productStatusTone('draft') to 'warning' (if drafts should read as needs-attention) or route all four surfaces through productStatusTone() so the Draft color is identical everywhere.
- **[MEDIUM · consistency]** Products.tsx command bar, search input, empty state (lines 733-794, 736-749, 993-1027)
  - Issue: The toolbar, search field, and empty state are hand-rolled even though canonical primitives exist: CommandBar, SearchInput (pill + clear + optional ⌘K), and EmptyState (tinted icon ring + title + actions). The bespoke search uses a squared bg-muted/50 Input while the app's SearchInput is a rounded-full pill with ring-focus — so Products' search looks different from Orders and other list pages.
  - Fix: Compose from ui-app: wrap the toolbar in <CommandBar>, swap the Input for <SearchInput value onValueChange shortcutHint>, and render both empty variants through <EmptyState icon={Package} .../> passing the action buttons. Removes ~40 lines of divergent markup and unifies the look.
- **[MEDIUM · usability]** useProductFilters.ts hasActiveFilters (lines 111-121) surfaced in Products.tsx (lines 841, 921)
  - Issue: hasActiveFilters returns true when only searchTerm or sortOption !== 'newest' is set. As a result the Filters button shows '(active)' and the results line shows '(filtered)' merely because the user changed sort order or typed a search — misleading, since sort is not a filter and neither lives in the filter panel the badge points at.
  - Fix: Split the concept: compute hasActiveFilters from category/tag/price/attribute/status only (what the filter panel controls), and track search/sort separately. Show the '(active)' pip on the Filters button only for real filter facets; reflect search/sort in the results summary text instead.
- **[MEDIUM · usability]** useProductData.ts error handling (lines 88-93, 131) + Products.tsx empty state (line 992)
  - Issue: A failed products fetch only console.errors and leaves allProducts empty with isLoading=false. With no cache, the page then renders the 'No products yet' empty state — so a network/permission error is indistinguishable from a genuinely empty catalog, and the user gets no retry.
  - Fix: Track an error flag in useProductData and render a distinct error state (EmptyState with an AlertTriangle tone + a 'Try again' button calling refetch(true)) when the fetch fails, instead of the onboarding empty state.
- **[LOW · consistency]** Products.tsx StatusPill component (lines 99-112) vs Orders STAT_PILL (Orders.tsx lines 96-114)
  - Issue: Products' status/filter pills use a neutral idle style (border-border bg-card) while the sibling Orders page uses tone-tinted idle pills (e.g. border-warning/25 bg-warning/5 text-warning). Same visual role, two different treatments. The counts also snap instantly with no count-up, unlike StatCard which supports useCountUp.
  - Fix: Adopt the Orders STAT_PILL tone recipe (idle tinted, active saturated) so the two pages match, and optionally animate the count with useCountUp from src/lib/anim for a subtle, on-brand refresh when stats change.
- **[LOW · cleanliness]** Products.tsx 'All' pill clear (line 799) + BulkActionsToolbar redundancy
  - Issue: Clearing the status filter via the 'All' pill is done by statusFilter.slice().forEach(handleToggleStatusFilter) — toggling each active status off one by one, which is fragile and non-obvious. Separately, status-setting logic (Active/Draft/Out of Stock icons+tones) is duplicated across BulkActionsToolbar, ProductStatusDropdown, and the table with independently hardcoded colors.
  - Fix: Add a dedicated clearStatusFilter() to useProductFilters and call it from the All pill. Extract a single status meta map (label, icon, tone) in lib/status and consume it in the dropdown, bulk toolbar, and pills so colors/icons never drift.
- **[LOW · modern]** ProductCard.tsx Card styling (lines 106-108)
  - Issue: Selected cards use shadow-2xl (very heavy drop shadow) plus ring-primary ring-offset-2, and selection-mode uses ring-2 ring-gray-400 — the harsh 2xl shadow and gray ring feel dated next to the app's soft shadow-sm/hover:shadow-md language.
  - Fix: Soften selected state to ring-2 ring-primary ring-offset-2 ring-offset-background with shadow-md (drop shadow-2xl); use ring-border/60 for the passive selection-mode outline. Keeps elevation consistent with StatCard/EmptyState.

## Product detail & editor (src/components/product-detail/* + src/components/ProductEditor.tsx)

_A functionally rich editor that still carries pre-design-system baggage: several surfaces hardcode raw emerald/amber/red palette (breaks the dark-safe token system), a 242-line dead VariantMatrix duplicate lingers, stock-editing is reimplemented twice with divergent controls, and heavy view-mode computation runs unmemoized on every keystroke — all fixable by adopting the existing ui-app primitives, status tokens, and GSAP reveal hooks._

- **[HIGH · modern]** VariantsManager.tsx stat tiles (lines 431-434) + variant stock dot (line 516)
  - Issue: The four inventory stat tiles hardcode text-emerald-600 / text-amber-600 / text-red-600 and the per-row stock dot hardcodes bg-emerald-500 / bg-amber-500 / bg-red-500. These raw-palette classes have no dark-mode variant, so they render wrong against the dark card background and violate the 'never hardcode raw palette' rule.
  - Fix: Route colors through the token system: use toneText/toneDotBg (or stockTone from src/lib/status.ts) instead of emerald/amber/red, and render the row indicator with <StatusDot tone={stockTone(inv)} />. Better yet replace the four bespoke tiles with the canonical <StatCard> (ui-app) using tone success/warning/danger/brand — it is token-based and dark-safe by construction.
- **[HIGH · modern]** ProductEditMode.tsx statusConfig (lines 31-35) + 'Find with AI' sparkle (line 396)
  - Issue: The status selector's statusConfig maps statuses to text-emerald-600 / text-amber-600 / text-slate-600, and the AI sparkle icon uses text-amber-400 — all raw palette, not dark-safe, and inconsistent with the StatusBadge tones used in ProductViewMode for the exact same statuses.
  - Fix: Drive the selector colours from productStatusTone()/toneText (src/lib/status.ts) so Active/Draft/Out-of-Stock match the view-mode badge, and swap text-amber-400 for text-primary or text-warning. Keep the lucide icons but colour them via tokens.
- **[HIGH · cleanliness]** VariantMatrix.tsx (entire 242-line file)
  - Issue: VariantMatrix is dead code — grep shows it is only referenced by itself and is never imported (VariantsManager superseded it). It is a divergent legacy implementation: Card/Table shell, a per-row RadioGroup 'Yes/No' default toggle, a leftover debug line 'Always visible when there are 2-3 option groups', and no token usage. It rots and misleads anyone reading the folder.
  - Fix: Delete src/components/product-detail/VariantMatrix.tsx. All variant editing already lives in the tokenized VariantsManager.
- **[HIGH · consistency]** ProductViewMode.tsx 'Manage Stock' modal (lines 357-486) vs VariantsManager variant table
  - Issue: Stock editing is implemented twice with different UX: the view-mode modal uses -/+ stepper buttons + a search/Filter Select and zebra rows, while VariantsManager (edit mode, Variants tab) edits the same product_variants.inventory via bare number inputs, a segmented status filter, chips and divide-y rows. Two visual languages and two code paths for one job (they even re-derive option_values from combination_key independently).
  - Fix: Consolidate onto one presentation. Either lift a shared VariantStockRow/table component used by both, or have the view-mode 'Manage Stock' modal reuse VariantsManager in a read-optimised mode. At minimum align the controls (pick stepper OR input) and the row styling (divide-y + hover, not zebra) so they feel like one product.
- **[HIGH · perf]** ProductViewMode.tsx render body (option-stock reduce lines 154-160; variants IIFE lines 215-245)
  - Issue: The option chips recompute derived stock with an O(values x variants) reduce for every value on every render, and the entire variants block is an inline IIFE that re-parses combination_key, filters, and sorts the full variant list on each render — including on every keystroke in the variant search box. convertCurrency is also called per option value per render. Nothing here is memoized.
  - Fix: Precompute a memoized map of option-value -> derived stock and a memoized processedVariants array with useMemo keyed on [variants, product.price, currency]; derive filtered/sorted inside a second useMemo keyed on the search/filter/sort state. This removes the per-keystroke full recompute.
- **[MEDIUM · usability]** ProductViewMode.tsx loading (isLoadingOptions spinner, lines 338-343)
  - Issue: While options/variants load, the hero renders immediately but the options block and the whole Variants section are simply absent, then pop in — and the only loading affordance is a lone centered Spinner appended AFTER the (empty) content, causing a visible layout shift when data arrives.
  - Fix: Render a Skeleton placeholder for the options row and the variants table region while isLoadingOptions (VariantsManager already does exactly this with ui/skeleton), so the panel keeps its shape and content fades in without a jump.
- **[MEDIUM · motion]** ProductViewMode.tsx (motion.div opacity, line 77) + ProductEditMode.tsx (line 290)
  - Issue: Both modes only cross-fade the whole panel via framer-motion opacity; sections (media/info hero, Specifications, Variants) all appear at once with no hierarchy or entrance polish. The app now ships a GSAP reveal system built for exactly this.
  - Fix: Add useReveal() from src/lib/anim to the scroll container and mark the hero, Specifications, and Variants blocks with data-reveal for a subtle staggered rise-in (reduced-motion aware). Keep it gentle per the productivity-app tone.
- **[MEDIUM · consistency]** VariantsManager.tsx stats grid (lines 429-441)
  - Issue: The inventory summary is a bespoke tile grid (border + text-lg + icon) that diverges from the canonical StatCard used elsewhere in the admin app, and its numbers change instantly with no count-up.
  - Fix: Replace the hand-rolled tiles with <StatCard> from ui-app (title/value/icon/tone). It brings token tones, hover elevation, and the reduced-motion count-up for free, matching sibling pages.
- **[MEDIUM · consistency]** Search fields: ProductViewMode variant toolbar (lines 262-265) & stock modal (lines 370-373); VariantsManager (lines 445-448)
  - Issue: Three separate places hand-roll an absolutely-positioned Search icon inside a raw Input at slightly different heights (h-7 vs h-8) and placeholders, instead of the canonical SearchInput pill used across Products/Orders.
  - Fix: Use <SearchInput value= onValueChange=> from ui-app for all three. It standardises the pill, focus ring, and adds a clear button, and removes the manual icon positioning.
- **[MEDIUM · consistency]** Empty states: ProductReviewsManager (line 102), ProductEditMode specs-empty (lines 414-419) & variants 'save first' (lines 437-441), VariantsManager options-empty (lines 377-383)
  - Issue: Four different bespoke 'nothing here' treatments — a bare centered <p> in reviews, two dashed boxes in the editor, and a custom icon+text block in VariantsManager — none use the shared EmptyState primitive, so tone, icon sizing, and spacing drift.
  - Fix: Swap each for <EmptyState icon= title= description= action=> (use compact for inline panels). e.g. reviews -> MessageSquare, options -> Layers with the 'Add your first option' button as action.
- **[MEDIUM · perf]** Auth waterfalls: ProductEditor.tsx handleImageUpload (line 148) & handleImageDelete (line 185); ProductEditMode.tsx handleFindSpecs (line 166); VariantsManager.tsx handleSave (line 289)
  - Issue: Four handlers call supabase.auth.getUser() — a network round-trip — just to read user.id, when AuthContext already exposes the user instantly from the cached session via useAuth(). This adds latency to every image upload/delete and save.
